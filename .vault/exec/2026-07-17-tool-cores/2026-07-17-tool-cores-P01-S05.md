# Prove live on the Claude lane that a document agent reads a named .vault ADR mid-turn and cites it, real run against the live engine with no mocks and zero .vault writes (executor-service)

## Scope

- `src/vaultspec_a2a/service_tests/`

## Description

- Add `test_tool_cores_floor_live.py` under `service_tests`, a live mock-free proof that a document agent reads a named `.vault` ADR mid-turn and cites it, with zero agent-origin `.vault` document writes.
- Extend the standing pw7 acceptance harness rather than fork a driver: reuse `_reachable_stack` for the infra gate and `AcceptanceHarness` (its `_mint`, `_run_start`) for token-mint and run-start against the live `vaultspec-adr-research` preset.
- Observe non-materializing: consume the run's public v1 SSE progress stream at `GET /v1/runs/{run_id}/stream`, witness the read and citation, then cancel the run before any review gate applies so no document materializes.
- Assert on the EMPIRICALLY-VALIDATED surface (see Outcome): the target ADR filename cited in a document agent's `message_chunk` stream, AND at least one distinctive interior token of the ADR body (an identifier/version/path present in the file but absent from the prompt) reproduced in that stream. A prompt-absent interior token can only reach the output by the agent reading the file, so it is the hallucination-resistant read evidence.
- Enforce zero writes with a before/after snapshot scoped to the vaultspec DOCUMENT directories (`adr`, `research`, `audit`, `plan`, `exec`, `reference`, `index`), excluding the engine-owned `.vault/data` runtime tree.
- Select the target ADR and derive its distinctive tokens from the live engine vault at run time (`_pick_named_adr`, `_distinctive_tokens`).

## Outcome

The floor is PROVEN live on the Z.ai lane and the harness is empirically validated. Z.ai rides the identical `AcpChatModel`/adapter as Claude (`factory.py`), so it validates both the harness and the shared floor code path. Green run: `pw7-1784274009` (2026-07-17, all-Z.ai profile) - a document agent cited the named ADR and reproduced its distinctive interior tokens (`@agentclientprotocol/claude-agent-acp`, `0.2.83`, `_KNOWN_MCP_SERVERS`, `registration-scope`, `36 releases`) with zero document-dir writes; the test PASSED in 79s. An earlier run `pw7-1784273382` captured the raw SSE frames (72 frames) that fixed two guessed assumptions.

Two surface corrections the live run forced, each now baked into the assertions:
1. The read does NOT surface as a `tool_call_start` frame (zero tool-call frames were emitted). It surfaces in `message_chunk` content, where the agent narrates the native discover-then-read sequence (tried `.vault`, found the file under `.vault/adr/`, read it) and reproduces the ADR's interior text. The assertion was moved off tool-call frames onto message content plus prompt-absent interior tokens.
2. A whole-`.vault` write watcher tripped on engine runtime churn only (`.vault/data/authoring-state/*.sqlite3-wal`, `.vault/data/engine-data/service.json` heartbeat) - not agent writes. The watcher was scoped to the document directories; the document surface stayed byte-identical across the run.

Grounding (rag-first): the `--type code` search "live service test drives research_adr run against live engine" surfaced `service_tests/harness.py` and `service_tests/test_pw7_acceptance.py` as the driver to extend; the observable-evidence trace went to the SSE frame schema (`api/schemas/events.py`), since `RunStatusResponse` carries no agent findings text.
