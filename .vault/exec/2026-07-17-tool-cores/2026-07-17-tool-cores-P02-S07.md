# Update the adapter entry-point resolution and npm install hint from the zed-industries path to the renamed agentclientprotocol package layout (executor-core)

## Scope

- `src/vaultspec_a2a/providers/factory.py`

## Description

- Repoint the node entry-point resolution scope directory in `factory.py` from `@zed-industries` to `@agentclientprotocol`; the `dist/index.js` bin layout is unchanged upstream, so only the scope segment moves.
- Update the missing-entry-point `ConfigError` install hint to name the renamed package.
- Update the two live package references in `service/docker/prod.Dockerfile` and `service/docker/README.md` to the renamed package.

## Outcome

The factory resolves the migrated node entry at `node_modules/@agentclientprotocol/claude-agent-acp/dist/index.js` and the resolved path exists on disk. A whole-tree sweep for the old name leaves two intentional non-dependency references untouched: the historical GitHub PR citation in `_acp_session.py` (`zed-industries/claude-agent-acp#380`, the accurate past location where the v0.20.2 terminal-auth gateway check landed), and the external `@zed-industries/codex-acp` mention in `knowledge/repositories/toad-acp-audit.md` (a reference audit of a different project's invocation, not our dependency). The existing factory test asserting `"claude-agent-acp" in cmd[0]` remains valid because the package short-name is unchanged. Gate: ruff clean, ty clean, all 22 factory tests pass.

## Notes

- The `_acp_session.py` citation was deliberately preserved rather than rewritten: PR #380 is a historical event in the old repository, and repointing it to the renamed repo would fabricate an unverifiable citation. The requirement it documents (terminal-auth `_meta` flags for gateway auth) still holds on 0.59.0.

- REVISION (reviewer HIGH-1, Docker cross-libc): the migration surfaced a build-time regression the host-side S08 regression could not catch. The renamed adapter bundles `@anthropic-ai/claude-agent-sdk@0.3.207`, whose native runtime ships as libc-SPECIFIC binaries (`claude-agent-sdk-linux-x64` for glibc, `…-linux-x64-musl` for musl, both `optional`/`os:linux`). `prod.Dockerfile` built `node_modules` in a `node:22-alpine` (musl) stage and copied them into the glibc worker (`python:3.13-slim-bookworm` + `node:22-slim`); with the old pure-JS SDK this was portable, but the new native binary installed under musl will not load under glibc. Fix: the `node-deps` stage was moved from `node:22-alpine` to `node:22-slim` (Debian/glibc, matching the worker), and the now-false "pure JS — portable" comments were corrected to state the real libc-matched-native-binary constraint. VERIFIED live in Docker (daemon 29.5.3): the `node-deps` stage built on glibc; the installed glibc `claude` binary executes under the worker's glibc runtime (`2.1.207 (Claude Code)`) and the adapter entry runs end to end (`0.59.0`); the CONTRAST musl variant fails under glibc with `exec: …/claude: not found` (the missing-musl-interpreter error), confirming the regression was real and the fix resolves it. Not verified: a full in-container Claude prompt turn (the full worker image build fails earlier at the unrelated `python-base` `uv sync` step resolving the `vaultspec-core` git dependency, an environment/network limitation independent of this change and of the ACP node runtime) - the node-runtime libc compatibility, which is the entirety of HIGH-1, was proven directly against the built glibc `node-deps` image.
