# Re-run the S20 registration-scope matrix on the migrated stack and record the decision-point outcome as an exec record: surfaced routes P03 to the existing composition path, not-surfaced routes P03 to the isolated-config-home surfacing fallback (executor-service)

## Scope

- `src/vaultspec_a2a/service_tests/`

## Description

- Discover the real vaultspec-rag MCP tool names by launching its stdio server through a live MCP client: `search_vault`, `search_codebase`, `get_code_file`, `reindex_vault`, `reindex_codebase`.
- Re-run the S20 registration-scope re-probe on the migrated stack: spawn the real migrated adapter subprocess, advertise the vaultspec-rag stdio launch spec from the known-server registry in the `session/new` `mcpServers` array (session-injected stdio), auto-permit the three read tool names via the `_meta.claudeCode.options.allowedTools` shape, and drive a real Claude turn instructing the model to invoke `search_codebase`.
- Run the re-probe under both the default deferred Tool Search path and the eager path (`ENABLE_TOOL_SEARCH=0`, the production condition whenever `allowed_tools` is present), so the deferred-tool registry cannot mask registered tools.
- Capture the adapter subprocess stderr and watch child processes to characterize connection versus surfacing.

## Outcome

DECISION-GATE VERDICT: NOT SURFACED.

On the migrated stack (`@agentclientprotocol/claude-agent-acp@0.59.0`, vendored `@anthropic-ai/claude-agent-sdk@0.3.207`, well past the 0.3.166 changelog entry that named a runtime-injection surfacing fix), a session-injected stdio MCP server still does not surface its tools to the model. Across every real turn the model never listed or invoked the injected tools and, when instructed to use `search_codebase`, took the explicit "no such tool" branch:

- Deferred path (default): the model issued two built-in `ToolSearch` tool calls, found nothing, and replied `NO_SUCH_TOOL`.
- Eager path (`ENABLE_TOOL_SEARCH=0`): the model issued zero tool calls and replied `NO_SUCH_TOOL` directly.

The strongest uncontaminated evidence that the injected config was accepted yet withheld from the model is the adapter's own SDK warning, emitted on the eager run:

  `[CLAUDE_SDK_CAN_USE_TOOL_SHADOWED] Warning: canUseTool will not be invoked for: mcp__vaultspec-rag__search_vault, mcp__vaultspec-rag__search_codebase, mcp__vaultspec-rag__get_code_file. Bare allowedTools entries auto-approve the whole tool before the callback is consulted.`

The SDK's permission layer parsed and named all three `mcp__vaultspec-rag__*` tools from our `allowedTools`, so the injected MCP server configuration reached the SDK and the tool identities registered there; the model's reasoning context nonetheless never received them. This is the exact S20 registration-scope gate (connected/configured but not surfaced; only user-global home-config servers surface), now reconfirmed on the latest adapter and SDK. The migration is still mandatory and correct (it lifts a dead, deprecated pin and closes the SDK security/behaviour gap), but it does NOT close the surfacing gate.

ROUTING CONSEQUENCE (per the plan's decision gate): NOT SURFACED routes P03 to the isolated-config-home surfacing fallback. Step P03.S14 must fire (the worker-owned isolated CLI config home additionally carries the declared read-only servers so they surface as user-global config), and the harness-provisioning invariant is amended per the ADR's conditional-amendment clause. The ambient-MCP suppression (P03.S13) is built regardless, as the ADR already mandates. The Claude and Z.ai semantic live proofs (P03.S16/S17) remain gated on S14, and the researcher-persona rag-tool naming (P03.S15) stays gated until surfacing is delivered by S14.

## Notes

- Methodology positive control: the same harness DOES surface tools the model can see — the model invoked the native built-in `Write` tool to a real filesystem side effect in the P02.S08 production turn and the native `ToolSearch` tool here. So the re-probe reliably detects surfaced tools; it is specifically the session-injected MCP surface that fails to reach the model. The "no such tool" result is therefore a true negative, not a broken harness or a prompt artifact (the prompt explicitly instructed tool use and offered the no-tool branch only as a fallback the model had to actively choose).
- Connection-vs-surfacing honesty: a `vaultspec-search-mcp` child process was observed during the run, but this machine hosts many concurrent vaultspec-rag processes from other worktrees and sessions (a standalone server on `:8766`, sibling `vaultspec-rag-worktrees` clients), so that observation cannot be cleanly attributed to this adapter and is NOT relied on as the connection proof. The connection half is instead established by the SDK-layer `SHADOWED` warning (the config demonstrably reached the SDK) and by the prior S20 record, which already proved session-injected stdio servers connect and serve. The dispositive new fact is the surfacing half, which is unchanged.
- Scope: the required minimum (session-injected stdio) was re-probed and is dispositive. Workspace-config registration was not separately re-probed; S20 recorded it as equally non-surfacing, and the SDK-layer evidence here (registration-scope gate, not transport or channel) predicts the same. If P03.S14's isolated user-global home surfaces as inferred, that is the positive path; if not, the residual gap is recorded as an explicit re-arm criterion per the plan's honesty limit.
- Live turns ran on the operator's logged-in Claude CLI credential reached via `CLAUDE_CODE_EXECUTABLE`; no token was written to disk. Turns were minimal single-instruction prompts.
