# Extend compose_harness_mcp_servers to accept and apply an allowlist so the composed servers exact tool names join the autonomous allowedTools, closing the attach-combined gap (executor-core)

## Scope

- `src/vaultspec_a2a/providers/_acp_mcp.py`

## Description

- Add a read-only `tools` tuple to the `vaultspec-rag` registry entry (`search_vault`, `search_codebase`, `get_code_file`) as allowlist metadata, deliberately omitting the server's `reindex_*` write verbs to honor the read-only composition boundary.
- Keep that metadata out of the ACP `session/new` payload: add a `_launch_spec` helper and have `resolve_harness_mcp_servers` return only the ACP-shape keys, so `tools` never leaks into the advertised mcpServer.
- Add `harness_allowed_tool_names`, mirroring `authoring_allowed_tool_names`: expand each declared server's read tools to the CLI's flat exact-name form `mcp__<server>__<tool>`, order-preserving, de-duplicated, and failing loudly on an unknown name.
- Extend `compose_harness_mcp_servers` with a keyword-only `allowed_tools` parameter that unions the passed names with the model's existing `allowed_tools` (never replacing) before calling `attach(combined, merged)`, closing the prior `attach(combined)` gap where composed servers were served but their tools never joined the autonomous allowlist. `None`/empty preserves the prior behavior exactly.

## Outcome

The composition seam can now auto-permit exactly the composed read-only rag tools. The union semantics are the load-bearing detail: `AcpChatModel.with_mcp_servers` REPLACES `allowed_tools` when a list is passed, so composing the harness AFTER the worker's authoring-attach step (which sets the authoring names) would have dropped the authoring names had compose passed a bare list; unioning with the model's existing `allowed_tools` preserves both sets. Existing behavior is unchanged when no allowlist is passed - the two current call sites (`worker.py`, `compiler.py`) still invoke the two-argument form and are untouched by this step (they are threaded in P03.S11). The read-only boundary is enforced at the registry: only the three search/read tools are ever expanded into the allowlist; the `reindex_*` write verbs are structurally excluded.

Gate: ruff clean, ty clean, 17 `test_acp_mcp` tests pass (11 prior plus 6 new covering the launch-spec metadata strip, the resolver expansion and its read-only exclusion, and the compose union - unchanged-without-allowlist, union-with-existing, and no-duplicate).

## Notes

- Grounding method (hybrid, per the standing directive): semantic-located the seam with `uvx vaultspec-rag search "autonomous mode allowed tool names allowlist for MCP composed servers" --type code` and `"authoring allowed tool names allowlist with_mcp_servers attach combined"` against the shared rag index, which returned the exact call chain (`worker.py` `attach(mcp_servers, allowed_tools)`, `_acp_authoring.authoring_allowed_tool_names`, `_acp_types` `allowed_tools` field); then confirmed the exact current symbols and the `with_mcp_servers` replace-not-merge semantics with grep and whole-file reads in the worktree. The p03 worktree's own rag index is empty, so the semantic half necessarily ran against the shared index (main's content), which is the correct dedup base anyway.
- Dedup: extended the existing registry/composition module and its existing `test_acp_mcp` suite rather than forking a parallel path; `harness_allowed_tool_names` deliberately mirrors the shape of the already-present `authoring_allowed_tool_names` rather than inventing a second convention.
- The three read tool names match exactly the three the P02.S09 SDK `CLAUDE_SDK_CAN_USE_TOOL_SHADOWED` warning enumerated, so the allowlist entries are grounded in observed adapter behavior, not guessed.
