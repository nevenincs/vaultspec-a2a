# Thread the composed rag tool names into the autonomous allowlist at the worker composition site alongside the authoring tool names (executor-core)

## Scope

- `src/vaultspec_a2a/graph/nodes/worker.py`

## Description

- At the worker composition site (after the authoring-attach step), import `harness_allowed_tool_names` alongside `compose_harness_mcp_servers` and, in autonomous mode only, resolve the declared harness servers' read tool names and pass them as `allowed_tools=` to compose. Supervised runs pass `None`, keeping their permission prompts. This mirrors the adjacent authoring-attach pattern (`allowed_tools = authoring_allowed_tool_names(binding) if autonomous else None`).
- Extend the existing `test_harness_mcp_wiring` protocol-layer suite (real `AcpChatModel` + ACP simulator recording `session/new` params) with two cases: a headless worker joins exactly the three rag read tool names to `_meta.claudeCode.options.allowedTools` (and no `reindex_*` write verb), and a supervised worker advertises the server but auto-permits none of its tools.

## Outcome

A headless document worker now auto-permits exactly the composed rag read tools, so once those tools surface (delivered by P03.S14) they invoke without stalling on a permission prompt. The change is a minimal additive edit at the single composition site: the compose call gains one keyword argument fed by an autonomous-gated resolver, and nothing else in the worker path moves. The union of the rag names with the authoring names happens inside `compose_harness_mcp_servers` (P03.S10), so the authoring bridge's allowlist entries are preserved rather than overwritten. Verified end to end at the protocol layer: the session the CLI is handed carries the three `mcp__vaultspec-rag__*` names in autonomous mode and carries none in supervised mode.

Gate: ruff clean, ty clean, 5 `test_harness_mcp_wiring` tests pass (3 prior plus the 2 new allowlist cases).
