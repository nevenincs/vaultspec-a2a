# Build the worker-owned isolated CLI config home that excludes the operator writable user-global MCP, delivering the harness ambient-MCP suppression required regardless of the re-probe outcome (executor-core)

## Scope

- `src/vaultspec_a2a/providers/acp_chat_model.py`

## Description

- Add a `_acp_config_home` module owning the per-run isolated CLI config home: `should_isolate_config_home` (gate), `create_isolated_config_home` (mkdtemp + a `.claude.json` carrying only onboarding flags, no servers), and `cleanup_isolated_config_home` (best-effort rmtree, never raises).
- Wire it into `AcpChatModel._astream`: when the spawn is the Claude/Z.ai path and auth is carried by an env token, create the home before spawn and set `CLAUDE_CONFIG_DIR` to it; clean it up in the existing `finally` after the subprocess is reaped.
- Add real-filesystem unit tests for the gate (token-present vs absent, gemini path, empty command), the created home's contents (onboarding flags present, no `mcpServers`), and cleanup idempotency.

## Outcome

Delivers the harness ambient-MCP suppression invariant, built regardless of the surfacing outcome. By redirecting `CLAUDE_CONFIG_DIR` to a fresh per-run directory that carries no credential file, the spawned CLI no longer resolves the operator's user-global `mcpServers` (the write-leak vector the invariant targets) NOR the operator account's connected remote MCP connectors: those connectors ride the operator's per-connector OAuth in `~/.claude/.credentials.json`'s `mcpOAuth` section, which is absent from the isolated home. Auth is unaffected because it rides the `CLAUDE_CODE_OAUTH_TOKEN`/`ANTHROPIC_AUTH_TOKEN` env, which is why the gate requires an env token - a run without one would depend on the operator's `~/.claude` credentials that the redirect removes, so it is deliberately left on the ambient home unchanged.

The suppression half was proven live on the migrated adapter (0.59.0, SDK 0.3.207) ahead of writing this code: with a redirected home carrying only onboarding flags plus a single declared server and token-env auth, the model listed and invoked exactly that server and did NOT see the operator's Figma/Gmail/Calendar/Drive account connectors (`model_mentions_figma: False`), where an un-redirected run lists all of them. The population that makes declared read-only servers SURFACE is P03.S14; this Step owns the isolation and writes no servers yet.

Gate: ruff clean, ty clean, 6 new `test_acp_config_home` unit tests pass, full non-service provider suite green (238 passed).

## Notes

- Scope boundary: the gate excludes the Gemini path (its own `GEMINI_CLI_HOME`) and only triggers on env-token auth, so existing service tests (bridge, migration-surface) that spawn the real adapter without a token are unaffected - isolation does not trigger and their behavior is unchanged. Confirmed: the two live-subprocess tests do not route through `_astream` (they drive raw spawn), and the simulator-backed harness-wiring tests carry no token.
- Lifecycle: the home is a `mkdtemp` dir cleaned in the `_astream` finally with `rmtree(ignore_errors=True)`, so a Windows file lock during teardown cannot raise; a leaked temp dir is a benign, self-identifying (`vaultspec-acp-home-*`) residue rather than a crash.
- Grounding: hybrid per the standing directive - the SDK config-resolution mechanism (`CLAUDE_CONFIG_DIR ?? ~/.claude`, credentials at `<dir>/.credentials.json`, `mcpOAuth` connector tokens) was read directly from the vendored `@anthropic-ai/claude-agent-sdk` bundle and the operator's real credential-file structure; the empirical suppression/surfacing behavior was established with real adapter subprocess probes before the production wiring was written.
