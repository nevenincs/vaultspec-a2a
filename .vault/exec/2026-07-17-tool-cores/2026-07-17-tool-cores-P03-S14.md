# If the P02 exec record shows session-injected servers do not surface, additionally populate the isolated config home with the declared read-only servers so they surface as user-global config (executor-core)

## Scope

- `src/vaultspec_a2a/providers/acp_chat_model.py`

## Description

- Add `config_home_mcp_servers` to `_acp_mcp`: from the session's advertised `mcp_servers`, keep only the registry-known harness servers (excluding the authoring bridge) and shape each into the CLI user-global `.claude.json` form keyed by name (`{type:stdio, command, args, env?}`).
- Extend `create_isolated_config_home` in `_acp_config_home` to accept the selected servers and write them as the home's `mcpServers`, so the CLI surfaces them as user-global config.
- Wire it in `AcpChatModel._astream`: when isolating, select the read-only harness servers and populate the home with them before redirecting `CLAUDE_CONFIG_DIR`.
- Add unit tests for the selection (registry-known only, `.claude.json` shape, env preservation, empty case) and the populated home; prove the full production path live.

## Outcome

FIRES UNCONDITIONALLY per the P02.S09 verdict (session-injected servers do NOT surface). The isolated config home now additionally carries the declared read-only harness servers, and they SURFACE to the model as user-global config - the ADR's contingency fallback, now the active path.

Proven live through the real production `AcpChatModel` path (not a hand-rolled probe) on the migrated adapter (0.59.0, SDK 0.3.207): a model constructed exactly as the composition produces it (vaultspec-rag in `mcp_servers` plus the three read-tool allowlist names) with token-env auth listed exactly the five `mcp__vaultspec-rag__*` tools and INVOKED `mcp__vaultspec-rag__search_codebase` mid-turn. The operator's account connectors (Figma/Gmail/Calendar/Drive) did NOT appear (`mentions_operator_connectors: False`), confirming S13 suppression holds through the production wiring. The `search_codebase` call reached the tool and returned the rag server's own "service is not running" error - a backend/infra condition (the rag service must be started for the search to return results), NOT a surfacing failure: the tool surfaced, was auto-permitted by the allowlist, and executed. Delivering actual search results is the executor-service live proofs (P03.S16/S17), which require the rag service up.

This resolves the ADR's second unproven inference in the affirmative: S20 proved only that the operator's REAL user-global home surfaces; this proves a REDIRECTED, populated home surfaces its declared servers. No STOP/ADR-re-decision was triggered.

Gate: ruff clean, ty clean, 247 non-service provider + harness tests pass (includes the new selection and populated-home cases). No import cycle (`acp_chat_model` -> `_acp_mcp` is acyclic).

## Notes

- Session-inject and config-home coexist without collision: the harness server is still advertised in `session/new` (from the composition) AND written into the config home, and the model saw exactly ONE set of rag tools - no duplicate registration or name clash. The session-injected copy remains inert (does not surface, per S09); the config-home copy is what surfaces. The redundant declaration is the ADR's "additionally populate" framing; if the reviewer prefers, the now-inert session-injection of registry-known harness servers could later be trimmed, but that is not required for correctness and is out of this Step's scope.
- Read-only boundary at the surface: the model listed `reindex_codebase`/`reindex_vault` as visible (declaring the rag SERVER surfaces all its tools; the CLI has no per-server tool filter in `.claude.json`, unlike Codex). They are NOT in the allowlist, so in headless mode they are not auto-permitted; a call would require a permission prompt that no human answers. Visible-but-not-auto-permitted is the same posture the ADR describes for Claude's flat allowlist. If strict non-visibility of write verbs is later required, that needs an upstream per-server tool filter this CLI does not expose.
- Auth/identity finding (de-risks production): the surfacing works with token-env auth and a home carrying ONLY onboarding flags plus the declared server - no `oauthAccount` seed and no credential file are needed. An earlier probe variant that authenticated via a copied credential file WITHOUT `oauthAccount` did not surface the local server, but the production token-env path does, so the CLI derives the account identity from the token. This is the true production configuration and is what the code uses.
- Grounding: hybrid per the standing directive - the SDK config-resolution internals and the `.claude.json` `mcpServers` shape were read from the vendored SDK bundle and the operator's real config; the surfacing/suppression behavior was established with real adapter subprocess probes, then re-proven through the production `AcpChatModel` wiring.
