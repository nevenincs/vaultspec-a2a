# Emit a per-run CODEX_HOME config.toml carrying the shared _KNOWN_MCP_SERVERS entries as mcp_servers blocks in the Codex config shape, one registry across two transports (executor-core)

## Scope

- `src/vaultspec_a2a/providers/codex_chat_model.py`

## Description

- Add `codex_mcp_server_specs` to `_acp_mcp`: resolve declared harness names to full read-only registry specs (`name`, `command`, `args`, `env`, `tools`) for the Codex transport, applying the same unknown-name and read-only fail-loud guards as the ACP path. Extract the shared `_require_read_only` guard so one trust root serves both transports.
- Add a `_codex_config_home` module (structural analog of `_acp_config_home`): `render_codex_config_toml` emits deterministic `[mcp_servers.<name>]` blocks (command/args, plus an `[mcp_servers.<name>.env]` sub-table when env is present); `build_codex_config_home` mkdtemps a per-run home, copies `auth.json` from the base home to preserve Codex file-based auth, and writes the config.toml; `cleanup_codex_config_home` best-effort removes it.
- Wire it into `CodexChatModel`: add a `harness_mcp_servers` field and, in `_astream`, build the per-run home from the declared servers, redirect `CODEX_HOME` to it, and clean it up in the existing `finally` after the subprocess is reaped.

## Outcome

Codex gains MCP grounding through the SAME `_KNOWN_MCP_SERVERS` registry as Claude, serialized to its native `config.toml` shape - one registry, two serializations. The per-run `CODEX_HOME` is the structural mirror of the Claude isolated config home: a worker-owned config dir carrying exactly the declared read-only servers, which suppresses the operator's ambient Codex MCP (the real `~/.codex/config.toml` on this machine declares `node_repl` and `computer-use` `[mcp_servers.*]` blocks - concrete suppression targets) while preserving auth via the copied `auth.json`. The emitted TOML shape was verified against the operator's real `~/.codex/config.toml` (`[mcp_servers.<name>]` table + `[mcp_servers.<name>.env]` sub-table) and every rendered document round-trips through stdlib `tomllib` in the tests. This Step emits the server blocks; the read-verb constraint (`enabled_tools`, `approval_mode`) is P04.S19.

Gate: ruff clean, ty clean; new deterministic tests pass - render/round-trip (rag block, env sub-table, empty), the registry accessor (read-only spec with tools, unknown-name raise, non-read-only fail-loud), and the home lifecycle (config written, `auth.json` copied, absent-auth tolerated, cleanup none-safe/idempotent). No live Codex turn was fired (deferred to executor-service S20/S21).

## Notes

- Module extraction disclosed: like the Claude side (`_acp_config_home` for plan-scoped `acp_chat_model.py`), the Codex serialization lives in a new `_codex_config_home` module wired into the plan-named `codex_chat_model.py`, for testability and symmetry with the ACP home. The shared registry accessor lives in `_acp_mcp` (the registry owner) despite its ACP-centric name, honoring the ADR's one-registry mandate rather than forking a second catalog.
- TOML writer choice: `tomlkit` is present but only as a transitive dependency (not in our `pyproject`), and `tomllib` is stdlib read-only, so the renderer is a small deterministic serializer (JSON string-escaping is a valid TOML basic-string subset for these values) and every emitted document is asserted parseable via `tomllib` - no reliance on an undeclared dependency, exact output under test.
- Auth is copied rather than env-carried because Codex auth is file-based (`auth.json` in `CODEX_HOME`), unlike the Claude token-env path; a base home without `auth.json` (env-based auth deployments) is tolerated with no copy.
- Credential-in-temp-dir security contract (the review's focus): the per-run home is created with `tempfile.mkdtemp`, which is owner-only (`0o700`) on POSIX, so the copied `auth.json` is traversal-protected by the directory even before its own mode is set; the copy is then explicitly pinned to `0o600` (POSIX-effective, best-effort `chmod` that is a no-op on Windows where the per-user temp tree is already ACL-scoped). A test asserts `0o600`/`0o700` on POSIX. Lifetime and cleanup on every catchable failure (HIGH-1 fix): the home is built INSIDE the `_astream` `try`, whose `finally` removes it with `shutil.rmtree(ignore_errors=True)`, so cleanup runs no matter where the turn fails - including a spawn failure, which the earlier structure (home built before the `try` opened) would have leaked with the credential inside. Two layers: `build_codex_config_home` also self-cleans its own dir if the build itself raises mid-copy (so a failed credential copy leaves no partial residue), and the outer `finally` covers every failure after a successful build. Tests exercise both: a copy-failure build self-clean and a real turn-failure (immediately-exited subprocess) that asserts no `vaultspec-codex-home-*` dir leaks. Hard-crash residue: on SIGKILL/power loss the `finally` cannot run, so the home (with the credential copy) leaks into the OS temp dir; the residue is bounded (one dir per interrupted turn), owner-only, and self-identifying by the `vaultspec-codex-home-*` prefix, and is reclaimed by normal OS temp cleanup - the same benign-leak posture as the Claude isolated home. Copy (not symlink) is deliberate: symlinked auth is not portable (Windows symlink privilege, and Codex may not follow it), and the copy's blast radius is contained by the owner-only short-lived dir.

- DISCOVERED DEFECT + FIX (post-landing, surfaced by the executor-service S21 live attempt): the config.toml emission built in S18/S19 was never wired to production - the preset's `[team.harness]` servers NEVER reached `CodexChatModel`. Trace: the worker/compiler composition site calls `compose_harness_mcp_servers`, which feature-detected `with_mcp_servers` and, finding none on the Codex model, silently RETURNED IT UNCHANGED; the factory's Codex branch never set `harness_mcp_servers`; so `_astream` always emitted the config.toml from an empty list. The MASKING-GAP LESSON: the S18/S19 deterministic tests set `harness_mcp_servers` DIRECTLY on the model, so they exercised the emission primitives in isolation and passed while the live composition path was dead - a test that constructs the object in a state production never reaches cannot catch a wiring break. Fix: give `CodexChatModel` an explicit `with_harness_mcp_servers` delivery method (parallel to `AcpChatModel.with_mcp_servers`) and have `compose_harness_mcp_servers` dispatch to it; ONLY a model with neither delivery mechanism (mock/hosted) is now returned unchanged - a model that HAS a harness mechanism can never be silently no-oped, the same fail-loud posture as the read-only trust-root guard. The regression test now builds the model through the REAL `compose_harness_mcp_servers` seam (not by setting the field) and asserts the emitted config.toml carries `vaultspec-rag` with the read-only `enabled_tools`; the config-home build was extracted to `CodexChatModel._build_codex_config_home` so the test exercises the production emission code rather than re-implementing it; and the remaining direct-field tests are annotated to state they cover defaults/lifecycle only, not the wiring. This landed on branch `tool-cores-codexwire` and must merge before the re-armed S21 live proof, or S21 would again exercise an unconfigured tool.
