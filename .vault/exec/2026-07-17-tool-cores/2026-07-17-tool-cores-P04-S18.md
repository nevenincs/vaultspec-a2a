# Emit a per-run CODEX_HOME config.toml carrying the shared _KNOWN_MCP_SERVERS entries as mcp_servers blocks in the Codex config shape, one registry across two transports (executor-core)

## Scope

- `src/vaultspec_a2a/providers/codex_chat_model.py`

## Description

- Add `codex_mcp_server_specs` to `_acp_mcp`: resolve declared harness names to full read-only registry specs (`name`, `command`, `args`, `env`, `tools`) for the Codex transport, applying the same unknown-name and read-only fail-loud guards as the ACP path. Extract the shared `_require_read_only` guard so one trust root serves both transports.
- Add a `_codex_config_home` module (structural analog of `_acp_config_home`): `render_codex_config_toml` emits deterministic `[mcp_servers.<name>]` blocks (command/args, plus an `[mcp_servers.<name>.env]` sub-table when env is present); `build_codex_config_home` mkdtemps a per-run home, copies `auth.json` from the base home to preserve Codex file-based auth, and writes the config.toml; `cleanup_codex_config_home` best-effort removes it.
- Wire it into `CodexChatModel`: add a `harness_mcp_servers` field and, in `_astream`, build the per-run home from the declared servers, redirect `CODEX_HOME` to it, and clean it up in the existing `finally` after the subprocess is reaped.

## Outcome

Codex gains MCP grounding through the SAME `_KNOWN_MCP_SERVERS` registry as Claude, serialized to its native `config.toml` shape - one registry, two serializations. The per-run `CODEX_HOME` is the structural mirror of the Claude isolated config home: a worker-owned config dir carrying exactly the declared read-only servers, which suppresses the operator's ambient Codex MCP (the real `~/.codex/config.toml` on this machine declares `node_repl` and `computer-use` `[mcp_servers.*]` blocks - concrete suppression targets) while preserving auth via the copied `auth.json`. The emitted TOML shape was verified against the operator's real `~/.codex/config.toml` (`[mcp_servers.<name>]` table + `[mcp_servers.<name>.env]` sub-table) and every rendered document round-trips through stdlib `tomllib` in the tests. This Step emits the server blocks; the read-verb constraint (`enabled_tools`, `approval_mode`) is P04.S19.

Gate: ruff clean, ty clean; new deterministic tests pass - render/round-trip (rag block, env sub-table, empty), the registry accessor (read-only spec with tools, unknown-name raise, non-read-only fail-loud), and the home lifecycle (config written, `auth.json` copied, absent-auth tolerated, cleanup none-safe/idempotent). No live Codex turn was fired (deferred to executor-service S20/S21).

## Notes

- Module extraction disclosed: like the Claude side (`_acp_config_home` for plan-scoped `acp_chat_model.py`), the Codex serialization lives in a new `_codex_config_home` module wired into the plan-named `codex_chat_model.py`, for testability and symmetry with the ACP home. The shared registry accessor lives in `_acp_mcp` (the registry owner) despite its ACP-centric name, honoring the ADR's one-registry mandate rather than forking a second catalog.
- TOML writer choice: `tomlkit` is present but only as a transitive dependency (not in our `pyproject`), and `tomllib` is stdlib read-only, so the renderer is a small deterministic serializer (JSON string-escaping is a valid TOML basic-string subset for these values) and every emitted document is asserted parseable via `tomllib` - no reliance on an undeclared dependency, exact output under test.
- Auth is copied rather than env-carried because Codex auth is file-based (`auth.json` in `CODEX_HOME`), unlike the Claude token-env path; a base home without `auth.json` (env-based auth deployments) is tolerated with no copy.
