# Constrain the Codex MCP surface to read verbs via enabled_tools with approval_mode auto for reads, keeping the read-only sandbox as defense-in-depth (executor-core)

## Scope

- `src/vaultspec_a2a/providers/codex_chat_model.py`

## Description

- Extend `render_codex_config_toml` so each `[mcp_servers.<name>]` block additionally emits `enabled_tools` naming exactly the registry's read tools for that server and `default_tools_approval_mode = "auto"`, constraining the Codex MCP surface to the read verbs and auto-approving them for headless runs.
- Add deterministic tests: `enabled_tools` equals the three rag read tools with no `reindex_*` write verb and approval mode `auto`; and a defense-in-depth test pinning that `CodexChatModel` keeps `approval_policy = "never"` and `sandbox = "read-only"`.

## Outcome

The Codex MCP surface is now read-only by construction on the more expressive Codex axis. `enabled_tools` is an exact allowlist naming only `search_vault`, `search_codebase`, `get_code_file` - the registry read tools that ride each spec from `codex_mcp_server_specs` - so even though the rag server also exposes `reindex_*` write verbs, they are never enabled for the agent. `default_tools_approval_mode = "auto"` lets those reads run without a prompt under the headless `approval_policy = "never"` composition. This layers over, not instead of, the existing `sandbox = "read-only"` jail (kept as-is on `CodexChatModel`), giving a stricter read-only posture than Claude's flat allowlist: the Codex agent's tool surface is trimmed to reads at the config layer AND the exec/filesystem path is sandboxed.

Gate: ruff clean, ty clean, full non-service provider suite green (255 passed) including the new read-verb constraint and sandbox-defaults tests; every emitted document round-trips through stdlib `tomllib`.
