# Prove live that a Codex document agent invokes vaultspec-rag search mid-turn under approval-policy never and sandbox read-only, capturing the tool-call trace and confirming citations resolve, real run with no mocks (executor-service)

## Scope

- `src/vaultspec_a2a/service_tests/`

## Description

- Drive the Codex semantic proof through the SAME harness as S17 (`test_document_agent_invokes_rag_search_midturn_and_cites`), profile-selected via `S05_PROFILE=codex-all` - no new driver. Intended to observe whether the rag MCP call is ADMITTED at runtime under Codex's `approval_policy:"never"` + `sandbox:"read-only"` + per-tool `default_tools_approval_mode="auto"` axis (the S19 config shape is proven deterministically; runtime admission was the open live-verify item).

## Outcome

BLOCKED on TWO independent findings; not proven, not faked, not worked around.

1. Codex usage limit (same as S20): the account is usage-limited until 2026-07-23 06:15 - a direct `CodexChatModel` app-server probe returned "You've hit your usage limit ... try again at Jul 23rd, 2026 6:15 AM." No live Codex turn is possible until then.

2. WIRING GAP (independent of usage; blocks S21 even with usage): the preset's declared harness servers are NEVER threaded into `CodexChatModel.harness_mcp_servers` in production, so the per-run `CODEX_HOME` config.toml (P04.S18) is always emitted from an EMPTY list - the Codex agent never receives the `vaultspec-rag` MCP server, so there is no tool to admit. Traced end to end: `compiler.py` computes `harness_mcp_servers = list(harness.mcp_servers)` and hands it to the worker; `worker.py` passes it ONLY to `compose_harness_mcp_servers`; that function (`_acp_mcp.py`) feature-detects `with_mcp_servers` and RETURNS THE MODEL UNCHANGED for any non-ACP model - Codex has no `with_mcp_servers`, so it silently no-ops. The factory's Codex branch (`factory.py`) constructs `CodexChatModel(...)` without `harness_mcp_servers` and does not read it from kwargs. Grep confirms no production site sets `CodexChatModel.harness_mcp_servers`; only the deterministic tests (`test_codex_config_home.py`, `test_harness_mcp_wiring.py`) construct it explicitly - which is why S18/S19 pass while the live path is dead. This precedes the admission-axis question: the tool is not even configured.

The admission-axis proof (does Codex invoke an MCP tool under approval_policy:never + sandbox:read-only) therefore CANNOT be answered as landed. It is gated on (a) a Codex harness-threading fix and (b) the usage window.

## Notes

Recommended fix (executor-core, Codex wiring owner): thread `harness_mcp_servers` into the Codex model - either the factory's Codex branch accepts and forwards it, or the worker/compiler sets it on a `CodexChatModel` via `model_copy` (parallel to how `compose_harness_mcp_servers` widens the ACP model). A focused test should construct the model through the SAME production path the worker uses and assert `harness_mcp_servers` is populated for a preset with a `[team.harness]` opt-in (the current deterministic tests bypass this by setting it directly).

Re-arm S21 (after the wiring fix lands AND Codex usage resets ~2026-07-23 06:15): boot engine (scoped to a workspace with real `.vault` + a code slice) + gateway; index that workspace on the running rag `:8766` service (`uvx vaultspec-rag index --port 8766`, attach); run `pytest -m service -k invokes_rag_search` with `S05_PROFILE=codex-all`; capture the run id, the agent's `mcp__vaultspec-rag__*` invocation, resolving citations, the rag daemon `POST /search` :8766 corroboration, and zero document-dir writes - and specifically confirm whether the MCP call is admitted under the composed approval axis (the S19 live-verify item).
