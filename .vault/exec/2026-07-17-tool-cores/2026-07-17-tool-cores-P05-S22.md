## Outcome

The landings had already absorbed the dead code the ADR named; this Step's residual work is framing corrections plus a documented keep-decision, with NO code deletion warranted. Per-target judgment:

- Allowlist-less `attach(combined)` in `_acp_mcp.py` (the S22 row's named target): NOT dead - it is the load-bearing supervised branch. P03.S10 already superseded the ADR's original always-taken allowlist-less gap by making the call conditional; the surviving `if not allowed_tools: return attach(combined)` is reached exactly when a supervised (non-headless) run declares harness servers (`allowed_tools=None`), advertising the servers without auto-permitting them so the human answers the permission prompt. It is exercised by `test_supervised_worker_does_not_auto_permit_harness_tools`. Removing it would break supervised harness composition, so it stays.

- `has_workspace_rules` (ADR-slated dead consumption): already absent from the tree - no matches - so nothing to remove, matching the plan's note that it was already resolved.

- The two stdio-surfaces-reliably residuals: corrected to state that neither session-injected transport surfaces on the current stack (S20 registration-scope matrix: only user-global home-config servers surface), so the stdio choice is a spawned-subprocess transport-mechanics decision, not a surfacing lever - matching the framing P01.S03/S04 already landed in `worker.py` and `_acp_authoring.py`.

Recommendation on the inert-looking session-inject copy (reviewer to judge at S24): KEEP, do not trim. It is NOT purely inert - `AcpChatModel._astream` reads the session-injected `mcp_servers` as the SOURCE that `config_home_mcp_servers` filters to populate the surfacing config home (P03.S14), so removing it would break surfacing. It is additionally the honest upstream re-arm channel: if a future CLI surfaces session-injected servers, that path lights up with zero code change. Its only cost is a redundant connect the CLI ignores on the current stack. Trimming just the redundant session/new advertisement (while keeping the data flow) would require separating "servers for session/new" from "servers for the config-home source" - a refactor with surfacing risk, out of a dead-code step's scope and not worth the redundant-connect saving. Implemented as a clarifying comment at the population site rather than a deletion.
