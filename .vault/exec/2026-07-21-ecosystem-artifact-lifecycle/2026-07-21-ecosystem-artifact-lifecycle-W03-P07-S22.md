# Add ignore rules for the generated artifacts that currently escape them

## Scope

- `.gitignore`

## Description

- Add rules for every generated artifact the audit found unmatched: the redirected
  process output suffix, coverage data and reports, distribution build output
  beside the already-ignored capsule tree, the continuous-integration cache
  directory, and packaging metadata.
- Write them outside the framework-managed block, which the framework regenerates.
- Verify each pattern matches a representative path.
- Verify no currently tracked file became ignored.

## Outcome

All seven artifact classes are now matched, each confirmed individually against a
representative path. Nothing currently tracked became ignored, checked by running the
whole tracked file list through the ignore rules and confirming an empty result - the
check that matters, since an ignore rule that shadows a tracked file produces confusing
behaviour long after the change.

The output-suffix rule is the one with history. The blanket rule covered the log suffix
but not the suffix a redirected process writes, which is how a large output file came to
sit in the machine-global home unmatched by anything. It never landed inside the
repository, so the exposure was latent rather than realised, and it is now closed.

## Notes

The rules were appended after the framework-managed block rather than inside it. That
block is regenerated by the framework and hand edits there are lost on the next
synchronisation; the audit that produced this Step recorded the ownership boundary, and
respecting it is the difference between a fix that survives and one that vanishes
silently.

The distribution rules are deliberately narrow, naming the wheel and archive suffixes
rather than the whole directory, so a future intentionally tracked artifact under that
tree is not swept up by a rule written today.
