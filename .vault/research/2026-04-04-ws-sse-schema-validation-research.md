# ws-sse-schema-validation research

Research into validating WebSocket and SSE event schemas that are not exposed
via the OpenAPI specification. The core problem: 12 ServerEvent variants and
6 ClientMessage variants are Pydantic models on the backend but hand-authored
TypeScript on the frontend, with no automated contract enforcement.

## Current state

The file `src/ui/src/app/data/wire-types.ts` is split into two sections:

- **Lines 1-1645**: auto-generated by `openapi-typescript` from the live
  OpenAPI spec (REST schemas only)
- **Lines 1647-1868**: hand-authored WebSocket protocol types mirroring
  Pydantic models in `src/vaultspec_a2a/api/schemas/events.py` and
  `commands.py`

The hand-authored section includes `ServerEvent` (12-variant discriminated
union on `type`), `ClientMessage` (5-variant union), plus component types
like `WsToolCallContent`, `WsPermissionOption`, `WsAgentSummary`,
`EventEnvelope`, and `ClientCommandBase`.

The backend has round-trip tests (`test_schemas.py`) that serialize and
deserialize every variant through `TypeAdapter(ServerEvent)` and
`TypeAdapter(ClientMessage)`, but there is **no automated check** that the
TypeScript definitions match.

## 1. AsyncAPI

### What it is

AsyncAPI is the event-driven counterpart to OpenAPI. Version 3.0.0 is the
current stable release. It defines channels, operations, and message schemas
for protocols like WebSocket, Kafka, AMQP, MQTT, and SSE.

### Tooling maturity

- **Spec**: AsyncAPI 3.0.0 is stable, adopted by companies like Slack,
  Salesforce, and SAP for documenting event-driven APIs.
- **Modelina** (`@asyncapi/modelina` v5.10.1): generates typed models from
  AsyncAPI/OpenAPI/JSON Schema specs. Supports TypeScript, Python, Java, Go,
  C#, and more. Can generate from AsyncAPI 2.0-3.0 specs.
  CLI: `asyncapi generate models typescript ./asyncapi.json`
- **AsyncAPI CLI** (`@asyncapi/cli`): validates specs, generates docs and
  code. Actively maintained.

### Python integration

- **`pydantic-asyncapi`** (PyPI, v0.3.0): Pydantic models representing the
  AsyncAPI 2.6.0 and 3.0.0 spec itself. Allows constructing an AsyncAPI
  document programmatically in Python. Does NOT auto-discover Pydantic models
  from your codebase; you must manually wire the schema references.
- **`asyncapi-schema-pydantic`**: similar but only supports AsyncAPI v2.
- **`asyncapi-eventrouter`** (Noteable): experimental framework that routes
  WS messages and generates AsyncAPI docs. Prototype quality, last updated
  2023.
- **Chanx** (v2.5.0): batteries-included WS framework for Django Channels
  and ASGI. Auto-generates AsyncAPI 3.0 docs from `@ws_handler` decorated
  consumers with Pydantic validation. **Django Channels specific**, not
  usable with raw FastAPI WebSocket endpoints.
- **`fastapi-asyncapi-docs`** (PyPI): adds an AsyncAPI docs endpoint to
  FastAPI apps. Very early stage.

### Assessment

AsyncAPI is the **right standard** for documenting our WS protocol, but the
Python tooling for FastAPI is immature. There is no "FastAPI AsyncAPI" that
auto-discovers WebSocket message schemas the way FastAPI auto-discovers REST
schemas. We would need to either:

- (a) Manually construct an AsyncAPI 3.0 spec using `pydantic-asyncapi`
  models, referencing `ServerEvent.model_json_schema()` output
- (b) Write a build-time script that generates the AsyncAPI spec from our
  Pydantic models

Either way, once we have an AsyncAPI spec, Modelina can generate TypeScript
types from it. The pipeline would be:
`Pydantic models -> AsyncAPI spec -> Modelina -> TypeScript types`

**Verdict**: high ceremony for our use case. The spec itself adds value for
documentation but the Python-to-AsyncAPI-to-TypeScript pipeline has too many
manual steps compared to the direct JSON Schema approach (see section 3).

## 2. Pydantic-to-TypeScript direct export

### Tools

- **`pydantic-to-typescript`** (v2.0.0, PyPI, Nov 2024): CLI tool that
  converts Pydantic models to TypeScript interfaces. Supports both Pydantic
  v1 and v2. Works by calling `model_json_schema()` internally, then piping
  through `json-schema-to-typescript`.
  Usage: `pydantic2ts --module my_module --output types.ts`
- **`pydantic2-to-typescript`** (fork): alternative fork, similar approach.

### Discriminated union handling

`pydantic-to-typescript` delegates to `json-schema-to-typescript` under the
hood. The critical question is whether the discriminated union `oneOf` +
`discriminator.propertyName` pattern that Pydantic v2 generates is correctly
translated to TypeScript discriminated unions.

Pydantic v2 generates discriminated unions as:

```json
{
  "discriminator": {
    "propertyName": "type",
    "mapping": {
      "agent_status": "#/$defs/AgentStatusEvent",
      "message_chunk": "#/$defs/MessageChunkEvent"
    }
  },
  "oneOf": [
    {"$ref": "#/$defs/AgentStatusEvent"},
    {"$ref": "#/$defs/MessageChunkEvent"}
  ]
}
```

`json-schema-to-typescript` (v15.0.4) has an **open feature request**
(issue #239) for explicit OpenAPI discriminator support. Without it, `oneOf`
schemas produce a plain TypeScript union (`A | B | C`) but do NOT narrow the
`type` field to a literal. This means the generated output would be:

```typescript
type ServerEvent = AgentStatusEvent | MessageChunkEvent | ...
// where AgentStatusEvent.type is string, not "agent_status"
```

This is **not** a proper TypeScript discriminated union. The `type` field
must be a string literal for TS narrowing to work.

### Assessment

`pydantic-to-typescript` is the simplest pipeline but produces unions that
lack literal discriminator narrowing. Post-processing or manual type
overrides would be needed to get proper TS discriminated unions.

**Verdict**: viable as a starting point but requires either:

- A post-processing step to narrow discriminator fields to literals
- A custom Pydantic JSON Schema mode that emits `const` instead of `enum`
  for Literal fields (Pydantic already does this for single-value Literals)

## 3. JSON Schema export + TypeScript generation (recommended path)

### The pipeline

```
ServerEvent.model_json_schema() -> JSON Schema -> codegen -> TypeScript
```

This is the most direct path with the fewest dependencies.

### What Pydantic v2 produces

For our `ServerEvent` union, `TypeAdapter(ServerEvent).json_schema()` emits
a JSON Schema with:

- Top-level `oneOf` array referencing each event variant via `$ref`
- `discriminator` object with `propertyName: "type"` and `mapping`
- Each variant's `type` field defined as `{"const": "agent_status"}` (for
  single Literal values) or `{"enum": ["agent_status"]}` depending on
  Pydantic version and config

The `const` keyword is the key: `json-schema-to-typescript` correctly maps
JSON Schema `const` to TypeScript literal types. So `{"const": "agent_status"}`
becomes `type: "agent_status"` in TypeScript.

### Known issues

- **Pydantic issue #7491**: nested discriminated unions can produce broken
  schemas. Our `ToolCallContent` union is nested inside `ToolCallStartEvent`
  and `ToolCallUpdateEvent`. This needs testing.
- **Pydantic issue #8628**: discriminated unions in nested dataclasses may
  produce `anyOf` instead of `oneOf`. Not a concern since we use BaseModel.
- **`$defs` resolution**: `json-schema-to-typescript` handles `$ref` and
  `$defs` correctly in current versions.

### Codegen tools comparison

| Tool | Version | Discriminator | Notes |
|------|---------|--------------|-------|
| `json-schema-to-typescript` | 15.0.4 | Partial -- reads `const`/`enum` on fields, ignores `discriminator` directive | Produces correct literals if Pydantic emits `const` |
| `quicktype` | 23.x | Maps `oneOf` to TS unions | Less control over output format |
| `openapi-typescript` | 7.x | Full discriminator support | Requires OpenAPI wrapper, not raw JSON Schema |

### Recommended approach

1. Export JSON Schema at build time:
   ```python
   from pydantic import TypeAdapter
   schema = TypeAdapter(ServerEvent).json_schema()
   # Write to src/ui/schemas/server-event.schema.json
   ```

2. Generate TypeScript with `json-schema-to-typescript`:
   ```bash
   npx json-schema-to-typescript \
     src/ui/schemas/server-event.schema.json \
     -o src/ui/src/app/data/ws-types.generated.ts
   ```

3. Validate in CI: re-export the schema, re-generate types,
   `git diff --exit-code`

**Verdict**: this is the lowest-ceremony, most reliable approach. The JSON
Schema is the single source of truth, the TypeScript is generated, and CI
catches drift.

## 4. Shared schema registries

### Options evaluated

- **Protobuf + Buf BSR**: Buf Schema Registry auto-generates SDKs for NPM,
  Go, Maven etc. Requires redefining all models in `.proto` files. Massive
  migration cost for our Pydantic-native codebase. Overkill.
- **TypeBox**: TypeScript-first JSON Schema builder. Wrong direction -- we
  need Python-first.
- **Effect Schema**: TypeScript-first, same problem.
- **Zod + zod-to-json-schema**: TypeScript-first.
- **JSON Schema as registry**: Using Pydantic's `model_json_schema()` output
  as the shared schema is effectively a "registry" without needing a separate
  tool. The schema files live in the repo and CI validates both sides.

### Assessment

All mature shared schema registries (Protobuf, Avro, Thrift) require
redefining models in an IDL, which duplicates our Pydantic definitions.
The JSON Schema approach keeps Pydantic as the single source of truth.

**Verdict**: use Pydantic's JSON Schema output as the de facto registry.
No external registry tooling needed.

## 5. CI contract validation patterns

### Pattern A: schema snapshot + codegen diff

The simplest and most widely used pattern:

1. At build time, export `ServerEvent.model_json_schema()` to a checked-in
   `.json` file
2. Generate TypeScript types from it
3. In CI, re-run both steps and `git diff --exit-code` -- any drift fails
   the build

This is used by projects like Stripe (OpenAPI spec committed, SDKs
generated) and Vercel (turborepo schema validation).

### Pattern B: runtime contract testing with Pact

Pact is the standard for consumer-driven contract testing. However:

- Pact's WebSocket support is experimental/nonexistent
- Pact is designed for HTTP request/response, not event streams
- Overkill for our single-consumer setup

### Pattern C: JSON Schema validation at both ends

- Backend: Pydantic validates outgoing events (already done)
- Frontend: validate incoming WS messages against the same JSON Schema
  using `ajv` (Ajv JSON Schema validator)
- CI: verify the schema files match between backend export and frontend
  bundle

### Pattern D: type-level compatibility check (TypeScript)

Export JSON Schema from Pydantic, generate a "reference" TypeScript file,
then use `tsc` to verify that the hand-authored types are assignable to/from
the generated types. This catches field additions, removals, and type
changes at compile time without runtime overhead.

### Recommended combination

**Pattern A + C**: schema snapshot in CI (catches drift at PR time) plus
optional Ajv validation in the frontend for runtime safety. Pattern A is
mandatory; Pattern C is a progressive enhancement.

## 6. FastAPI WebSocket schema exposure

### Current state

FastAPI does NOT expose WebSocket endpoint schemas in its OpenAPI output.
The OpenAPI 3.1 spec itself has no native WebSocket support -- it is
HTTP-centric. FastAPI's `@app.websocket("/ws")` decorator creates a route
but contributes nothing to `/openapi.json`.

### Available workarounds

- **Custom OpenAPI schema extension**: override `app.openapi()` to inject
  `x-websocket-events` or similar custom extensions into the spec. This
  makes the schema available at `/openapi.json` but standard tooling
  (`openapi-typescript`) ignores unknown extensions.
- **Separate schema endpoint**: serve the WS JSON Schemas at a dedicated
  endpoint like `/api/schemas/ws-events`. This is what we should do for
  development tooling.
- **AsyncAPI companion spec**: maintain a separate AsyncAPI spec alongside
  the OpenAPI spec. This is the "proper" solution but adds maintenance
  burden (see section 1).
- **OpenAPI 4.0 (Moonwalk)**: the next major OpenAPI version has proposals
  for event-driven API support, but it is not yet released.

### Assessment

There is no clean way to get WS schemas into the OpenAPI spec that standard
tooling understands. The separate JSON Schema file approach (section 3)
sidesteps this entirely.

**Verdict**: do not try to force WS schemas into OpenAPI. Export them as
standalone JSON Schema files and generate TypeScript from those.

## Recommendation

The recommended approach for this project, ordered by implementation
priority:

1. **Export JSON Schema from Pydantic at build time** -- a small Python
   script that calls `TypeAdapter(ServerEvent).json_schema()` and
   `TypeAdapter(ClientMessage).json_schema()`, writing to
   `src/ui/schemas/server-event.schema.json` and
   `src/ui/schemas/client-message.schema.json`

2. **Generate TypeScript from JSON Schema** -- use
   `json-schema-to-typescript` (already in the npm ecosystem). Verify
   that `const` fields produce literal types for proper TS narrowing.

3. **CI validation** -- a CI step that re-exports schemas and re-generates
   types, then `git diff --exit-code` to catch any drift between Pydantic
   models and generated TypeScript.

4. **Optional: Ajv runtime validation** -- add `ajv` to the frontend to
   validate incoming WS messages against the schema at runtime (development
   mode only, stripped in production).

5. **Optional: AsyncAPI spec** -- if external documentation of the WS
   protocol is needed, generate an AsyncAPI 3.0 spec from the same Pydantic
   models using `pydantic-asyncapi`. This is documentation, not enforcement.

### What NOT to do

- Do not adopt Protobuf or any external IDL -- Pydantic is the source of
  truth
- Do not use Pact for WS contract testing -- it lacks WS support
- Do not try to inject WS schemas into OpenAPI -- the spec does not support
  it
- Do not adopt Chanx -- it is Django Channels specific
- Do not rely on `pydantic-to-typescript` alone -- it lacks discriminated
  union literal narrowing without post-processing

### Critical validation needed before ADR

Before committing to the JSON Schema pipeline, one experiment must be run:

```python
from pydantic import TypeAdapter
from vaultspec_a2a.api.schemas.events import ServerEvent
import json
schema = TypeAdapter(ServerEvent).json_schema()
print(json.dumps(schema, indent=2))
```

Verify that:

- Each variant's `type` field uses `{"const": "agent_status"}` not
  `{"enum": ["agent_status"]}`
- The nested `ToolCallContent` discriminated union inside
  `ToolCallStartEvent.content` is correctly represented
- All `$ref` paths resolve within the `$defs` block
- `StrEnum` values (e.g. `ToolKind`, `AgentLifecycleState`) are emitted as
  `{"enum": [...]}` with all values listed

Then feed the schema through `json-schema-to-typescript` and verify the
output produces proper TypeScript literal types for discriminator fields.
