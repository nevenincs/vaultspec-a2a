# `multi-provider-execution` research: `provider matrix, ACP env-injection seam, and per-role assignment for Codex, Claude, and Z.ai across the LangGraph pipeline`

Grounds the owner's target — mixing Codex, Claude, and Z.ai (Z.ai routed through Claude Code) as providers inside `research_adr` and future graph topologies, with per-role assignment and cross-repo initialization — in the current provider stack (`src/vaultspec_a2a/providers/`), the model-profiles precedence chain (`[[model-profiles-adr]]`), and the adr-authoring-orchestration graph-submitter topology (`[[adr-authoring-orchestration-adr]]`). Gemini is grounded only where it illustrates an existing pattern; it stays out of scope per the owner's framing.

## Findings

### Provider matrix

**(a) Claude-direct — proven, unchanged.** `ProviderFactory.create` (`src/vaultspec_a2a/providers/factory.py:353-393`) spawns `claude-agent-acp` (node entry or Bun binary, `_classify_acp_command`, `factory.py:184-226`) as an `AcpChatModel` (`providers/acp_chat_model.py`), injecting only `CLAUDE_CODE_OAUTH_TOKEN` (never `ANTHROPIC_API_KEY`, which is explicitly popped when the OAuth token is present — ADR-002 §2, `acp_chat_model.py:253-258`). This path is live-tested (model-profiles P03.S05) and unaffected by anything below.

**(b) Z.ai-via-Claude-Code — a same-code-path config variant, not a new integration.** The `claude-agent-acp` wrapper itself defines the hook for this: `GatewayAuthMeta` (`knowledge/repositories/claude-agent-acp/src/acp-agent.ts:171-182`), read from `session/new`'s `_meta` field (`acp-agent.ts:475`), is converted by `createEnvForGateway` (`acp-agent.ts:1404-1415`) into exactly `ANTHROPIC_BASE_URL`, `ANTHROPIC_CUSTOM_HEADERS`, and an empty `ANTHROPIC_AUTH_TOKEN` — the wrapper's own comment states this is "to redirect API calls via baseUrl / inject custom headers / bypass default Claude login." Independently, the spawned process's env is built as `{...process.env, ...userProvidedOptions?.env, ...createEnvForGateway(...)}` (`acp-agent.ts:1242-1246`), so plain env passthrough also works.

On the Python side, the equivalent injection point already exists and is provider-neutral: `AcpChatModel._astream` builds `env = resolve_env_vars(_ws_path)` then `env.update(self.env_vars)` (`providers/acp_chat_model.py:251-252`) — `self.env_vars` is the provider-supplied dict `ProviderFactory.create` constructs (today just `{CLAUDE_CODE_OAUTH_TOKEN: ...}` for Claude, `_build_gemini_env` for Gemini at `factory.py:41-66`). `workspace/environment.py:70-89`'s scrub list does **not** include `ANTHROPIC_BASE_URL` or `ANTHROPIC_AUTH_TOKEN` (it does unconditionally scrub `ANTHROPIC_API_KEY`, `environment.py:72`), so a Z.ai variant only needs a `_build_zai_env(base_url, auth_token)` helper mirroring `_build_gemini_env`, wired into a new provider branch in `factory.py`'s dispatch — the subprocess plumbing (`_subprocess.py`, `_acp_session.py`) is untouched. This is a provider **config variant** of the existing Claude ACP path, exactly as the owner framed it, not a new code path.

Open and unverified: whether Z.ai's Anthropic-compatible endpoint implements the full surface `claude-agent-acp` depends on (tool-calling schema fidelity, streaming chunk shape, any Anthropic-specific headers/beta flags the CLI sends) closely enough for the ACP session handshake and tool-permission bridge to behave identically. No evidence either way was found in this repo or the vendored wrapper source; this needs a live probe against the real Z.ai endpoint, not an assumption from "Anthropic-compatible" marketing language.

**(c) Codex — not ACP; a different protocol needing a different chat-model class.** Codex is not one protocol. As an HTTP target it is the OpenAI Responses API (`knowledge/repositories/CLIProxyAPI/internal/runtime/executor/codex_executor.go:36`: "CodexExecutor is a stateless executor for Codex (OpenAI Responses API entrypoint)"). As a local CLI runtime it is `codex app-server`, a genuine JSON-RPC-over-stdio server — confirmed in this machine's installed codex plugin (`C:\Users\hello\.claude\plugins\cache\openai-codex\codex\1.0.4\scripts\lib\app-server.mjs:189`, spawning `codex app-server` and framing `id`/`method`/`params` over stdin/stdout, lines 81-158, 189-241, 331). The companion `codex-companion.mjs` script — the thing the owner's brief refers to as "the codex plugin whose companion runtime routes through Claude Code" — drives this JSON-RPC client itself; it does not make Codex speak ACP or route model calls through Claude Code's own model. "Routes through Claude Code" describes the **plugin's invocation surface** (a Claude Code skill/slash-command shells out to the companion script), not a protocol bridge A2A could reuse to make Codex ACP-compatible.

Codex's JSON-RPC shape (`initialize`, id/method/params framing, notification handler) is structurally similar to ACP but is Codex's own bespoke protocol. Adapting it to ACP's `session/new`/`session/prompt`/`session/update` vocabulary would duplicate what the codex-companion plugin already does, for no gain. The cheaper, contract-clean path is a dedicated `BaseChatModel` subclass that either shells out to the companion script or drives `codex app-server` directly with its own JSON-RPC client — bypassing `AcpChatModel`/`_acp_session.py` entirely. `providers/mock_chat_model.py` is the existing precedent for exactly this shape: a `BaseChatModel` with no ACP handshake, dispatched in `factory.py:348-351` alongside the ACP and `ChatOpenAI`-backed branches (Zhipu, OpenAI, `factory.py:440-491`). `_subprocess.py`'s process-lifecycle helpers (`spawn_acp_process`/kill-tree) are protocol-agnostic and reusable for Codex's subprocess management even though its wire protocol is not ACP's.

### Per-role provider assignment: the precedence chain exists, but fan-out branches share one provider

The model-profiles ADR's precedence chain (profile > worker override > agent TOML > team defaults) is implemented once in `resolve_role_assignment` (`providers/model_profiles.py:184-253`) and consumed uniformly by the compiler, discovery, and run-start. `TeamProfileRoleConfig` (`team/team_config.py:290-301`) overlays provider/capability/fallback **per worker `agent_id`**, and the `vaultspec-adr-research` preset's four roles (researcher, synthesist, adr-author, doc-reviewer — `team/presets/teams/vaultspec-adr-research.toml:84-94`) already demonstrate a working per-role profile (`[team.profiles.fast]`, same file lines 102-115) that varies capability by role today. Extending it to vary **provider** by role (researcher=codex, synthesist=claude, adr-author=zai) requires no new mechanism — a `TeamProfileRoleConfig.provider` overlay per `agent_id` already resolves correctly through the existing chain, PROVIDED each of those roles maps to a distinct `agent_id`/worker.

The real gap is the **diverge fan-out**: `_compile_research_adr` resolves exactly one `BaseChatModel` per role name via `_resolve_research_adr_models` (`graph/compiler.py:942-953`, consumed at `:1085-1091`) and passes the single `models["researcher"]` instance to `_make_research_producer` for every parallel branch (`graph/compiler.py:1093-1098`, branch specs from `team_config.topology.research_threads`, `team/team_config.py:319-329`). All N researcher branches (`research_dispatch_researcher_00`, `_01`, ... — confirmed node names in `graph/tests/test_research_adr.py:96-127`) share the same provider instance. Assigning genuinely different providers to different fan-out branches (e.g. one research thread on Codex, another on Claude, for provider-diversity research) is not possible today without extending `ResearchThreadSpec` (`team/team_config.py:319-329`) with its own model overlay and threading it through `_resolve_research_adr_models`/`_make_research_producer` per-branch rather than per-role. This is a genuine schema gap, not a bug — flagging for the ADR's decision.

### Cross-repo initialization

The frozen a2a-edge contract (`[[a2a-edge-conformance-adr]]`, `[[a2a-edge-conformance-reference]]`) is v1-additive: `run-start` takes a preset id, prompt, target feature, and actor-token bundle; `presets-list` and the model-profiles discovery record (`model-profiles-plan` P02.S03) already serve `provider` as a plain string per role/profile (not observed anywhere as a closed wire-level enum on the A2A side — `Provider` is an internal `StrEnum`, `graph/enums.py:128-135`, serialized as `.value` in `FrozenAssignment`/`RoleEligibility`, `providers/model_profiles.py:499-554`). Adding `Provider.ZAI` and `Provider.CODEX` members is additive on A2A's side by the same mechanism that already lets `zhipu`/`openai`/`mock` ride the existing discovery and run-start surface without a prior contract amendment: no new verb, no renamed field, no run-start payload shape change.

What is **not verified in this repo** and must not be assumed: whether the dashboard/engine's own schema for consuming `provider`/profile fields treats it as an open string (compatible) or a closed enum requiring its own update before it accepts `zai`/`codex` values without validation failure. That is dashboard-repo state this research cannot ground — it is an explicit open question for the ADR, not a decision this feature can make unilaterally.

### Coordination: the graph is provider-agnostic, with two Claude-CLI-specific exceptions

The LangGraph compiler and `research_adr` topology consume only `BaseChatModel` instances (`_resolve_research_adr_models` return type, `graph/compiler.py:949`) and read back the writer's document from `AIMessage.name` stamped by the worker node (matched against `PhaseAuthoringSpec.writer_message_name`, `authoring/submitter.py:98-119`) — both are LangChain/graph-level contracts with no provider-specific assumption. Two real provider-specific behaviors were found, both confined to the ACP path:

- `ENABLE_TOOL_SEARCH=0` (`acp_chat_model.py:269-280`) is a workaround for the pinned Claude CLI's deferred-tool-registry behavior over a stdio MCP bridge; it is inert for non-ACP or non-Claude-CLI providers. Z.ai inherits it for free (same CLI binary); Codex needs its own MCP-bridging mechanism entirely, since it never goes through `AcpChatModel`.
- The OAuth-token / `ANTHROPIC_API_KEY` mutual-exclusion logic (`acp_chat_model.py:253-263`) is Claude-CLI-specific billing behavior; a Z.ai variant uses `ANTHROPIC_AUTH_TOKEN` (a different variable) so this logic does not need to change, but should be reviewed to confirm it does not misfire when both a Z.ai token and a stale OAuth token are present in the environment simultaneously.

Not investigated (flagged, not assumed): per-provider capability/tool-calling-shape parity for Z.ai specifically (see above); Codex's actual authentication model for non-interactive/headless invocation (ChatGPT login vs. API key vs. local session) and whether it can run credential-free the way `codex app-server` is typically invoked interactively; provider-level cost/quota signals for either new provider (out of scope, mirrors the model-profiles ADR's own readiness-vs-quota distinction).
